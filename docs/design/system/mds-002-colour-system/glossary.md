# Glossary

---

# Purpose

This glossary defines the terminology introduced by **MDS-002 — Colour System**.

Unlike previous glossaries, these definitions describe architectural colour concepts rather than implementation values.

Every future MDS specification should use these definitions consistently.

Changing colour terminology changes how contributors reason about the Design System.

For that reason, these definitions should remain stable.

---

# A

## Adaptive Neutral Foreground

A light or dark calibrated neutral text or icon colour selected from local resolved luminance.

The resolver uses hysteresis so Composition movement does not cause rapid switching near a contrast threshold.

---

## Acrylic Tint Recipe

One of the internal Clear, Mist, Smoke or Deep Smoke neutral transmission recipes.

Recipes change neutral pigmentation and transmission without changing Acrylic physics.

---

## Acrylic Tint Intent

Semantic content intent that allows the Colour System to select an Acrylic Tint Recipe without exposing the recipe or Material coefficients to the consumer.

## Accessibility Colour

A resolved colour that satisfies accessibility requirements while preserving semantic meaning.

Accessibility Colours are generated through Theme Resolution rather than manually authored.

---

## Atmosphere

The adaptive or static environmental lighting generated from current artwork or a governed brand illumination pair.

Atmosphere is derived from artwork when artwork is available.

It influences Materials.

It never replaces Brand or Semantic Colours.

---

## Atmosphere Token

A Semantic Colour whose Resolved Token value is influenced by governed environmental context.

Examples include:

- Atmosphere.Primary
- Atmosphere.Reflection
- Atmosphere.Highlight

The Semantic Colour roles are Platform-authored and their Resolved Token values are generated.

---

# B

## Brand Colour

A stable colour representing the Mosaic identity.

Brand Colours communicate:

- recognition
- consistency
- product identity

They intentionally do not communicate runtime emotion.

---

## Brand Illumination Pair

The default Indigo and Cyan pair, or a resolved co-brand pair containing Mosaic Indigo and one normalised registered partner accent, used as a static environmental light source when no focused or Hero artwork is available.

The pair influences Acrylic through the Material System rather than recolouring components directly.

---

# C

## Co-Branding

Governed partner presence through an approved mark and Brand Illumination Pair while Mosaic remains the primary product identity.

Co-Branding does not permit white-labelled themes or partner-owned Design Tokens.

---

## Co-Brand Pair Resolver

The deterministic Platform process that scores registered partner colours against Mosaic Indigo, normalises the selected accent within governed limits and produces a safe Brand Illumination Pair.

---

## Colour Collision

A condition in which no registered partner candidate can form a perceptually distinct, accessible and Material-safe pairing with Mosaic Indigo without unacceptable brand-identity loss.

---

## Colour Resolution

The deterministic process that transforms Semantic Colour intent into resolved implementation values.

Colour Resolution considers:

- Theme
- Accessibility
- Runtime Atmosphere
- Device

before producing final colours.

---

## Colour Synthesis

The runtime process responsible for generating environmental atmosphere from entertainment artwork.

Colour Synthesis influences Materials.

It never changes semantic meaning.

---

# L

## Light Source

The conceptual origin of Runtime Atmosphere.

Within Mosaic, artwork is treated as a light source rather than a colour palette.

This distinction underpins the Refraction System.

---

# M

## Material Colour

The colour produced after Runtime Atmosphere has interacted with a Material.

Material Colours are generated.

They are not authored.

---

# P

## Partner Brand Registration

A governed co-brand record containing one required signature colour and optional approved alternatives, preference order and usage restrictions.

It provides candidates to the Co-Brand Pair Resolver rather than a complete partner-owned theme or illumination pair.

## Primitive Colour

A measurable physical colour value.

Examples include:

- Cyan 500
- Slate 950
- White
- Black

Primitive Colours intentionally contain no semantic meaning.

---

# R

## Reflected Colour

A colour generated by the interaction between Runtime Atmosphere and Materials.

Reflected Colours should appear subtle and environmental.

They should never dominate semantic hierarchy.

---

## Runtime Atmosphere

The adaptive environmental colour model generated from the user's current World.

Runtime Atmosphere represents emotional lighting rather than interface colouring.

---

# S

## Semantic Colour

A colour role representing meaning rather than implementation.

Examples include:

- Surface.Primary
- Text.Secondary
- Action.Primary
- Status.Success

Applications should consume Semantic Colours rather than Primitive Colours.

---

## Surface Colour

A Semantic Colour describing structural hierarchy.

Surface Colours communicate:

- depth
- grouping
- organisation

They should not communicate interaction or status.

---

# T

## Theme

A deterministic interpretation of Semantic Colours.

Themes implement:

- Light
- Dark
- OLED
- Accessibility

without changing semantic meaning.

---

## Theme Resolution

The process by which Semantic Colours become theme-specific implementation values.

Theme Resolution occurs before Runtime Atmosphere.

# Terminology Rules

Future contributors should:

- describe meaning before colour values
- describe atmosphere before implementation
- distinguish Brand from Atmosphere
- distinguish Semantic Colour from Primitive Colour
- avoid implementation-specific terminology inside architectural documentation

Colour terminology should remain independent from rendering technology.
